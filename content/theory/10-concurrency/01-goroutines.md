# Горутины

Горутина — лёгкий поток выполнения, управляемый планировщиком Go. Это фундамент конкурентности в Go.

## Запуск горутины

```go
go функция(аргументы)
```

```go
func main() {
    go fmt.Println("из горутины")
    fmt.Println("из main")
    time.Sleep(time.Millisecond) // дать горутине выполниться
}
```

Вывод программы (возможный вариант):
```
из main
из горутины
```

Порядок строк не гарантирован — горутина и main выполняются независимо, и «из горутины» может напечататься как до, так и после «из main».

Анонимная функция:

```go
go func() {
    fmt.Println("анонимная горутина")
}()  // обязательно вызвать немедленно через ()
```

С замыканием:

```go
msg := "hello"
go func() {
    fmt.Println(msg)  // захватывает msg
}()
```

## Планировщик Go: M:N threading

Go использует модель M:N threading: множество горутин выполняются на небольшом количестве OS-потоков. Планировщик Go сам решает, какую горутину запустить на каком потоке.

Представь три уровня:

- **Горутины** — лёгкие задачи, которые создаёт твой код. Их может быть тысячи.
- **OS-потоки** — реальные потоки операционной системы. Их немного (обычно равно числу ядер).
- **CPU-ядра** — физическое железо, которое всё это выполняет.

```
G1  G2  G3  G4  G5  G6   ← горутины (много, лёгкие)
 \   |   /   \   |   /
  T1     T2     T3        ← OS-потоки (мало, тяжёлые)
   |      |      |
  CPU1  CPU2  CPU3        ← ядра процессора
```

Планировщик Go ставит горутины в очередь и раскидывает их по потокам. Если горутина ждёт (например, IO) — поток не простаивает, а берёт следующую горутину из очереди. Поэтому 10 000 горутин спокойно работают на 4 потоках.

**Преимущества:**
- Горутины стартуют с маленьким стеком (~2-8 KB), который растёт по мере необходимости
- Создание тысяч горутин — нормально (в отличие от OS-потоков, где каждый ~1-8 MB стека)
- Кооперативное + преемптивное планирование (Go 1.14+)

### GOMAXPROCS

Количество OS-потоков для выполнения горутин. По умолчанию = количество CPU-ядер:

```go
import "runtime"

fmt.Println(runtime.GOMAXPROCS(0))  // текущее значение
runtime.GOMAXPROCS(4)               // установить вручную

fmt.Println(runtime.NumCPU())       // количество физических ядер
fmt.Println(runtime.NumGoroutine()) // текущее количество горутин
```

## Goroutine leak — утечка горутин

Горутина «течёт» когда она запущена, но никогда не завершится — просто висит в памяти и ждёт чего-то, что никогда не случится.

Одна-две застрявших горутины — не катастрофа. Проблема возникает в долго живущих сервисах: каждый HTTP-запрос запускает горутину, горутина застревает, и за несколько часов их накапливаются тысячи. Это приводит к:

- **Росту памяти** — каждая горутина держит стек (от 2 KB) и все захваченные переменные. 10 000 утёкших горутин — это легко несколько гигабайт.
- **Утечке ресурсов** — горутина может держать открытый файл, соединение с БД или сетевой сокет. Они не освободятся, пока горутина не завершится.
- **Деградации производительности** — планировщику приходится обслуживать тысячи горутин, которые ничего полезного не делают.
- **OOM kill** — в какой-то момент ОС убивает процесс из-за нехватки памяти, сервис падает.

### Частые причины утечек

**Чтение из канала, который никогда не закрывается:**

```go
// УТЕЧКА: горутина ждёт вечно
func leaky(ch <-chan int) {
    go func() {
        for v := range ch {  // если ch никогда не закроют...
            fmt.Println(v)
        }
    }()
}
```

**Запись в канал без читателя:**

```go
// УТЕЧКА: горутина заблокирована навсегда
func leaky2() {
    ch := make(chan int)
    go func() {
        ch <- 42  // никто не читает
    }()
}
```

**HTTP-запрос без контекста:**

```go
// УТЕЧКА: горутина выполняет запрос даже после отмены операции
func startRequest() {
    go func() {
        resp, _ := http.Get("https://slow-api.example.com/data")
        // обработка...
        _ = resp
    }()
}
```

### Как обнаружить утечку

**runtime.NumGoroutine() в тестах:**

```go
func TestNoLeak(t *testing.T) {
    before := runtime.NumGoroutine()
    
    // ... запускаем код ...
    doSomething()
    
    time.Sleep(10 * time.Millisecond) // дать время завершиться
    after := runtime.NumGoroutine()
    
    if after > before {
        t.Errorf("горутин утекло: было %d, стало %d", before, after)
    }
}
```

**Пакет goleak (golangci-lint):**

```go
import "go.uber.org/goleak"

func TestMain(m *testing.M) {
    goleak.VerifyTestMain(m)
}
```

### Как предотвратить утечки

**1. Done-канал для отмены:**

```go
func worker(done <-chan struct{}) {
    for {
        select {
        case <-done:
            return  // корректное завершение
        default:
            doWork()
        }
    }
}

done := make(chan struct{})
go worker(done)
// ...
close(done)  // сигнал завершения
```

**2. context.Context:**

```go
func worker(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            return
        default:
            doWork()
        }
    }
}

ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
go worker(ctx)
```

> 💡 context.Context подробно разберём в главе 12 «Продвинутые темы».

## sync.WaitGroup — ожидание горутин

Самый распространённый способ дождаться завершения группы горутин:

```go
import "sync"

func main() {
    var wg sync.WaitGroup
    
    for i := 0; i < 5; i++ {
        wg.Add(1)  // увеличить счётчик перед запуском горутины
        go func(i int) {
            defer wg.Done()  // уменьшить счётчик при завершении
            fmt.Println("горутина", i)
        }(i)
    }
    
    wg.Wait()  // ждать пока счётчик не станет 0
    fmt.Println("все горутины завершились")
}
```

**Правило**: всегда вызывай `wg.Add(1)` **до** запуска горутины, не внутри неё.

## Горутины и замыкания: ловушка

Замыкание захватывает переменную по ссылке, а не по значению. Если переменная меняется после запуска горутины — горутина увидит новое значение.

```go
// i объявлена снаружи цикла — все горутины смотрят на одну переменную
i := 0
for ; i < 5; i++ {
    go func() {
        fmt.Println(i)  // напечатает 5 5 5 5 5 — цикл уже завершился
    }()
}
```

К моменту когда горутины начнут выполняться, цикл уже закончился и `i` равна 5 — все горутины напечатают одно и то же.

Исправление — передать значение явно через параметр:

```go
i := 0
for ; i < 5; i++ {
    go func(n int) {
        fmt.Println(n)  // 0 1 2 3 4 (в произвольном порядке)
    }(i)  // i копируется в n в момент запуска горутины
}
```

> Обрати внимание: если написать `for i := 0; i < 5; i++` — Go 1.22+ создаёт новую переменную `i` на каждой итерации, и проблемы нет. Ловушка актуальна когда переменная объявлена снаружи цикла.

## Итог

- `go f()` запускает f в новой горутине; управление немедленно возвращается
- Планировщик Go: M горутин на N OS-потоках (M:N threading)
- Горутины легковесны — создавай тысячи, не OS-потоки
- GOMAXPROCS = количество параллельных потоков (по умолчанию = кол-во CPU)
- Горутина течёт, когда никогда не завершается: используй done-канал или context
- `sync.WaitGroup` — для ожидания группы горутин; `Add` перед запуском, `Done` в defer
