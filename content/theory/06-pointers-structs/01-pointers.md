# Указатели

Указатели в Go проще, чем в C/C++: нет арифметики указателей, автоматическая сборка мусора, строгая типизация. Но понимание указателей необходимо для правильного написания методов и работы с большими структурами.

## Основы

**Указатель** — переменная, хранящая адрес другой переменной в памяти.

- `&` — взять адрес переменной (address-of оператор)
- `*` — разыменовать указатель (dereference, получить значение по адресу)
- `*T` — тип "указатель на T"

```go
x := 42
p := &x  // p — указатель на x, тип *int

fmt.Println(p)   // 0xc0000b4000 — адрес в памяти
fmt.Println(*p)  // 42 — значение по этому адресу

*p = 100  // изменяем значение через указатель
fmt.Println(x)  // 100 — x изменился!
```

### Нулевое значение указателя — nil

```go
var p *int  // нулевое значение для любого указателя = nil
fmt.Println(p)  // <nil>

// Разыменование nil — паника!
// fmt.Println(*p)  // panic: runtime error: invalid memory address or nil pointer dereference
```

Всегда проверяй указатель на nil перед разыменованием:

```go
func printValue(p *int) {
    if p == nil {
        fmt.Println("nil pointer")
        return
    }
    fmt.Println(*p)
}
```

## new() — создание указателя

`new(T)` выделяет память для нового значения типа T, инициализирует нулевым значением и возвращает `*T`:

```go
p := new(int)   // *int, *p == 0
*p = 42
fmt.Println(*p)  // 42

s := new(string)  // *string, *s == ""
*s = "hello"

// Для структур:
type Point struct{ X, Y int }
pt := new(Point)  // *Point, pt.X == 0, pt.Y == 0
pt.X = 10
```

На практике `new` используется редко — чаще используют литералы структур с `&`:

```go
pt := &Point{X: 10, Y: 20}  // эквивалентно new + инициализации
```

## Указатели в функциях

Передача по значению vs по указателю:

```go
// По значению — функция получает копию:
func doubleValue(x int) {
    x *= 2  // изменяем копию
}

// По указателю — функция может изменить оригинал:
func doublePtr(x *int) {
    *x *= 2  // изменяем оригинал
}

n := 10
doubleValue(n)
fmt.Println(n)  // 10 — не изменился

doublePtr(&n)
fmt.Println(n)  // 20 — изменился!
```

### Когда передавать по указателю

1. **Нужно изменить оригинал**: как `doublePtr` выше
2. **Большая структура**: копирование дорого по памяти и времени
3. **Опциональное значение**: указатель может быть nil (аналог `Option<T>` в Rust)

## Value Semantics vs Pointer Semantics

Это одна из ключевых концепций дизайна Go-программ.

### Value Semantics (семантика значения)

Копия при передаче. Безопасно. Изолировано:

```go
type Point struct{ X, Y float64 }

func scale(p Point, factor float64) Point {
    p.X *= factor
    p.Y *= factor
    return p  // возвращаем новую копию
}

original := Point{3, 4}
scaled := scale(original, 2)

fmt.Println(original)  // {3 4} — не изменился
fmt.Println(scaled)    // {6 8}
```

### Pointer Semantics (семантика указателя)

Разделение состояния. Изменения видны всем держателям указателя:

```go
type Counter struct{ count int }

func (c *Counter) Increment() {
    c.count++
}

c := &Counter{}
c.Increment()
c.Increment()
fmt.Println(c.count)  // 2
```

### Когда какую использовать

| Выбор | Когда |
|-------|-------|
| Value | Маленькие структуры (до ~64 байт), неизменяемые данные |
| Pointer | Большие структуры, изменяемое состояние, необходимость nil |

## Указатели и сборка мусора

В Go есть сборщик мусора (GC). Ты не управляешь освобождением памяти вручную — GC освободит память, когда на объект не останется ссылок:

```go
func makeString() *string {
    s := "hello"   // создаётся на стеке или куче
    return &s       // компилятор «эскейпит» s на кучу, так как возвращаем адрес
}

p := makeString()
fmt.Println(*p)  // hello — s жива, пока p ссылается на неё
// когда p выйдет из scope, GC освободит память
```

Компилятор Go делает **escape analysis** — автоматически решает, где разместить переменную: на стеке (быстро, но ограниченно) или на куче (медленнее, но неограниченно).

## Типичные ошибки

**Ошибка 1**: Разыменование nil.

```go
var p *User = getUser() // может вернуть nil
p.Name  // паника если p == nil

// Правильно:
if p != nil {
    fmt.Println(p.Name)
}
```

**Ошибка 2**: Возврат указателя на локальную переменную — ОК в Go (не как в C).

```go
func newInt(x int) *int {
    return &x  // БЕЗОПАСНО: Go эскейпит x на кучу
}
// В C это был бы dangling pointer, в Go — нет
```

**Ошибка 3**: Ожидать изменения через копию указателя.

```go
type Node struct{ Val int }

func wrongUpdate(n *Node) {
    n = &Node{Val: 99}  // изменяем локальную копию указателя!
}

node := &Node{Val: 1}
wrongUpdate(node)
fmt.Println(node.Val)  // 1 — не изменился!

// Для изменения оригинала:
func correctUpdate(n *Node) {
    n.Val = 99  // изменяем значение через разыменование
}
correctUpdate(node)
fmt.Println(node.Val)  // 99
```

## Итог

- `&x` — адрес переменной x, возвращает `*T`
- `*p` — значение по адресу p
- Нулевое значение любого указателя — `nil`, разыменование nil — паника
- `new(T)` — выделить память, инициализировать нулём, вернуть `*T`
- Передача по указателю позволяет изменять оригинал и избегать копирования
- Go делает escape analysis — не нужно вручную управлять памятью
- Разница между изменением `*p` (оригинал) и `p` (локальная копия указателя)
